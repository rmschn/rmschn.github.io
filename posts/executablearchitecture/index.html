Executable Architecture

Architecture is often thought of as a design phase that happens before any code is written. However, architecture can and should be executable. What I mean by that is a design can be coded, that design code can be translated into IaC, that IaC can be deployed in various environment levels (dev, test, staging, production). Beyond this the architecture can be managed and automated like a software project. Here I am going to attempt to outline how this might be done, because I have been talking about it for years and want to share the idea. Its likely hardly new, but possibly not widely thought about because it requires a significant shift in how systems architecture and design are approached.

Design as Code

The first step is to treat architecture as code. This means that the design of the system is written in a language, such as d2lang. This design code can be thought of as a blueprint for the system, and it can be version controlled, tested, and automated just like any other code. This allows for a more iterative and collaborative design process. You can automate the generation of diagrams. You can also automate the validation of the design against certain rules or constraints. For example, you could have a rule that all services must have a certain level of redundancy, and you could automate the checking of the design against that rule.

Design to IaC

The next step is to translate the design code into Infrastructure as Code (IaC). Or if you started with IaC you could generate the design from the IaC. IaC can be done using tools like Terraform, CloudFormation, or Pulumi. This allows for the design to be directly implemented in the infrastructure, and it also allows for the infrastructure to be easily replicated across different environments.

Governance

Once the architecture is executable, it can be governed and managed like any other software project. This means that you can have a CI/CD pipeline for your architecture, where changes to the design code trigger automated checks and balances ensuring say, that the design adheres to the policies and non-functional and functional requirements of the organization. In other words you can have a growing repository of approved designs that can be easily reused and adapted to different projects.

This also allows for better collaboration between architects and developers, as they can work together on the design code and see how it translates into the actual infrastructure. It also allows for better communication with stakeholders, as they can see the design in a more tangible way and understand how it will be implemented.

Conclusion

So far as I know, there is no one doing this quite this way. With the introduction of AI, and the increasing availability of tooling that can be pipelined together and partnered with traditional development tools, this is becoming more and more possible. I think this is the future of architecture, and I am excited to see how it evolves. It will require a significant shift in how we think about architecture and design, but I believe it will lead to better systems and better collaboration between architects and developers.
